 1  """ 
 2  Zen is designed to be a Zero-Effort Network library - meaning that whether you just want to prototype code or need to  
 3  implement a high-speed routine, Zen makes it possible for you to do it easily.  In order to acheive this, Zen has made 
 4  use of several specific architectural features which enable (1) ease in learning and using the library and (2) the ability 
 5  for the library to offer maximally fast routines. 
 6   
 7  Most fundamentally, every node and edge in a Zen graph has three properties: 
 8   
 9          - an index: this is a unique integer which is assigned to the entity.  In the underlying implementation, 
10                  this index provides a very fast lookup to information about the object, which is important when 
11                  writing code where execution time is important.  Because this index is a direct lookup, this value 
12                  may not be perserved across instances of the graph. 
13          - an object: this is a higher-level identifier for the entity.  Nodes can be given arbitrary objects, so 
14                  long as they are hashable.  In ordinary directed and undirected graphs, the identifier for an edge is 
15                  its endpoints (ordered endpoints, in the case of a directed graph).  The object for a node and edge 
16                  must be both hashtable and immutable - at least with regard to the features that determine its  
17                  hashcode and comparison to other objects. 
18          - data: this is an object (often a dictionary) that contains attributes of the node or edge. 
19           
20  When operating on nodes and edges in a graph, they can be referred to using either their index or their object. 
21  Except where it wouldn't make sense, every method that accepts nodes or edges as input has two versions - one  
22  that takes an object and one that takes an index.  Methods that use an index will **always** be faster than one  
23  that takes an object.  In order to facilitatevreadability and easy learning of method and function names, everywhere  
24  in Zen we use the following conventions: 
25   
26          - If a specific function accepts a node or edge, then the version that accepts objects will be called 
27                  "<fxn>(...)".  The one which takes the index will be called "<fxn>_(...)". 
28                   
29          - Any function that is called "<fxn>(...)" and returns an edge or node will return an object. Any  
30                  function that is called "<fxn>_(...)" and returns an edge or node will return an index. 
31  """
